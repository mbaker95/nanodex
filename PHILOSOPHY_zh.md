# NanoDex 理念

NanoDex 是 NanoClaw 在 Codex 生态中的延续版本，目标是在适配 Codex 原生工作流的同时，尽可能忠实保留 NanoClaw 的理念与使用体验。

## 核心原则

### 足够小，足够清晰

NanoDex 应保持在一个人可以理解宿主运行时、容器运行时、存储模型、设置流程和通道行为的规模之内，而不需要平台团队或复杂的运维体系。

### 通过真实隔离保障安全

智能体应运行在具有明确挂载边界的容器中。安全性应来自真实的系统级隔离，而不是应用内部的权限戏法。

### 为个人 Fork 而生

NanoDex 的目标不是演化成通用平台或托管控制面板，而是成为一个你可以 Fork、阅读、修改并持续塑造成个人助手的代码库。

### 代码优先于配置

少量简单的环境变量是可以接受的，但大部分行为变化都应通过代码或指令文件表达，而不是不断叠加新的配置层。

### Agent 就是界面

安装、调试和定制都应通过 Codex 在仓库内部完成。NanoDex 不应依赖 shell 向导、仪表盘或旁路配置系统来完成核心体验。

### AGENTS.md 是记忆模型

共享与分组记忆应保存在 `AGENTS.md`：

- `groups/global/AGENTS.md`：共享指令
- `groups/<group>/AGENTS.md`：群组级指令
- 工作区文件：长期保留的局部事实和笔记

### Skills 优先于功能膨胀

当某些通信路径确实属于首启核心体验时，可以直接内置在仓库中。除此之外，新能力应尽量通过 repo skills 或对 Fork 的直接修改来实现，而不是不断膨胀基础运行时。

### WhatsApp 是规范默认路径

最忠于 NanoClaw 的解释，是把 WhatsApp 视为默认产品路径。

其他通道可以存在，甚至可以为了改善首启体验而直接随仓库提供，但它们应始终被视为次级选项，而不是与默认路径同等地位的产品核心。

### 运行时必须前后一致

`npm start`、bootstrap、设置流程、实际运行行为以及仓库文档，必须共同指向同一种体验。如果 NanoDex 说“Agent 是界面”，那代码也必须这样运行。

## NanoDex 是什么

NanoDex 是：

- 一个小型 Node.js 宿主编排器
- 一个基于 Docker 的 Codex 运行时
- 一个按群组隔离工作区和会话状态的个人助手
- 一个可以在你自己的 Fork 中不断演化的代码库

## NanoDex 不是什么

NanoDex 不是：

- 通用自动化平台
- 多租户托管服务
- 以仪表盘为中心的控制平面
- 一个默认就试图内置所有集成的运行时

## 对新改动的标准

一个好的改动，应该至少在不破坏其他原则的前提下，改善以下某一项：

- 更容易理解
- 更容易信任
- 更容易让 Codex 安全修改
- 更忠于 NanoClaw 的精神和体验
- 更符合个人 Fork 的使用方式

如果某个改动让 NanoDex 在精神、设置流程或运行模型上更不像 NanoClaw，那么除非有非常强的理由，否则应将其视为回归。
